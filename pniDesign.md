# pniDesign process

BPMN2 process **`GC-AUTOMATION.pniDesign`** implementing the `PNIDesign`
workflow from `workflow_steps_v02 CE 30 Mar.xlsx` (rows 6-20, column D =
`PNIDesign`).

- BPMN file: `src/main/resources/com/gc/gc_automation/pniDesign.bpmn`
- OpenAPI contract: `/pniDesign` in
  `210_automaton/automaton-data/static/cross_wf_api_0.97.yaml`
- Matching request/response schemas: `PniDesignRequest` / `PniDesignResponse`

## Where it sits in the pipeline

```
ODIN ──POST /pniDesign──▶ Automaton (Camel + Groovy route)
                              │
                              ├─▶ store request in automaton.tmf_store (source=GC-AUTOMATION)
                              │
                              └─▶ wfService.startWfSynchronousProcess(
                                      "GC-AUTOMATION.pniDesign",
                                      { _order: <tmf_store.id> })
                                              │
                                              ▼
                                   jBPM (this kjar) executes pniDesign.bpmn
                                              │
                                              ├─▶ SdcApiCall tasks post to
                                              │      http://.../api/jbpm/apiCall
                                              │   (URL injected as _apiCallUrl global)
                                              │
                                              ├─▶ handler reads routeMap.domain
                                              │      + routeMap.source and dispatches
                                              │      to the matching "sources" entry
                                              │      in Automaton's runtime-config.js
                                              │
                                              └─▶ each source then calls
                                                     CROSS Core API
                                                     (POST /v1/project/, /v1/service/, …)
```

The process is *synchronous*: the response Map it produces is returned as
the HTTP body to ODIN (same pattern as `resourceAvailabilityCheck`).

## Inputs (`PniDesignRequest`)

| Field | Type | Notes |
|-------|------|-------|
| `version` | string | API contract version, echoed back in the response |
| `addressMasterId` | string | Global reference id of the customer address |
| `edgeEquipmentCrossId` | string | Cross id of the MAS/MER chosen by the previous `resourceAvailabilityCheck` call |
| `serviceId` | string | External service identifier (ODIN-assigned) |
| `crossProjectName` | string | Desired CROSS project name, typically the ODIN order id |
| `productId` | string | Service template identifier — drives which service components get created |

Process variable `_order` is the `tmf_store.id` generated by Automaton
before the jBPM process is started — that id is how the TMF_STORE task
retrieves the original request body from tmf_store.

```
  │                              │

┌────────────────────────────────┐

└────────────────────────────────┘

                        │    createLocality    │      localitySmwId

┌──────────────────────────────────┐  3D.1  find free cage or transceiver
│ 7. cross/findEdgeEquipmentCage   │  on the selected edge equipment
└──────────────────────────────────┘  → cageCrossId OR edgeTransceiverCrossId

  │                                     equipmentName, cageName, portName,
  │                                     portCrossId, CageCrossId,

└────────────────────┘

  │                                  → opticalPathCrossId, opticalPathPrmId

┌────────────────────────────────┐  3F  create service from service template
│ 10. cross/createService        │  template = productId, id = serviceId

  │                                   PRODUCT1, ACCESS, PHYSICAL_ACCESS
  │                                   → serviceCrossId
  ▼
┌────────────────────────────────────────┐  3G.1  attach OP + locality to service

┌───────────────────────────┐   3H  build PniDesignResponse Map
│ Create result: COMPLETED  │   (see "Response" below)
└───────────────────────────┘

```

The error path is the same shape as `resourceAvailabilityCheck`:

```
any "Failed" throw-link event
      │
      ▼
  catch link event ──▶  Create result: FAILED  ──▶  end
                        (puts errorMessage
                         into response.error)
```

Gateway condition expressions set `errorMessage` via
`kcontext.setVariable("errorMessage", "…")` before routing to the Failed
throw event, so the FAILED script always has a message to put in the
response body. Today only the address gateway participates in the error
path — other failure points throw RuntimeException inside the handler,
which you'd catch and rewire later if desired.

## Response (`PniDesignResponse`)

Assembled in the COMPLETED script task from the process variables:

```json
{
  "id":                 "<_order>",
  "version":            "<version>",
  "status":             "COMPLETED",
  "serviceCrossId":     "<serviceCrossId>",
  "OpticalPathCrossId": "<opticalPathCrossId>",
  "OpticalPathPrmId":   "<opticalPathPrmId>",
  "localityCrossId":    "<localityCrossId>",
  "localitySmwId":      "<localitySmwId>",
  "projectCrossId":     "<projectCrossId>",
  "edgeEquipment": {
    "edgeEquipmentCrossId": "…",
    "equipmentName":        "…",
    "cageName":             "…",
    "CageCrossId":          "…",
    "portName":             "…",
    "portCrossId":          "…"
  }
}
```

The `edgeEquipment` sub-object is returned as-is by
`cross/findEdgeEquipmentCage` — i.e. that source is expected to emit
exactly the shape defined in the `EdgeEquipment` schema in
`cross_wf_api_0.97.yaml`, so no reshaping happens in the BPMN.

## What was built in this kjar

- `pniDesign.bpmn` — 12 SdcApiCall tasks, 2 script tasks, 3 exclusive
  gateways, start + 2 end events, 2 throw-link + 1 catch-link events,
  24 sequence flows. Build passes with `mvn clean install`
  (`kie-maven-plugin` validates the BPMN at package time).
- Process id: `GC-AUTOMATION.pniDesign` in package
  `com.gc.gc_automation`. Version `1.0`, executable, public, synchronous.
- `.gen_pniDesign.py` — the generator script used to template the BPMN
  so UUID prefixes stay consistent across `itemDefinition` /
  `dataInput` / `dataInputAssociation` references (the thing
  `CLAUDE.md` warns about if you rename tasks by hand). Regenerate by:

  ```bash
  python3 .gen_pniDesign.py
  ```

## What was **NOT** built (owned by other repos / manual steps)

1. **SVG — `GC-AUTOMATION.pniDesign-svg.svg`.** The BPMN's diagram
   section (BPMNDI) has a flat auto-generated layout only. The
   preferred workflow is to open `pniDesign.bpmn` in Business Central,
   tidy the layout, and save — the modeler regenerates the SVG.
   Commit the `.bpmn` and `.svg` together, per repo convention.

2. **Automaton `runtime-config.js` sources.** The BPMN references
   these `routeMap.source` names; each must be added as a `sources[]`
   entry (mirroring the existing `cross/executeQuery/findAddress` /
   `cross/executeQuery/findEquipments` examples). Without them the
   handler has no URL to call and the task fails.

   | `routeMap.source` | Purpose | Expected output |
   |-------------------|---------|-----------------|
   | `cross/createProject` | POST CROSS `/v1/project/` | `projectCrossId` |
   | `cross/executeQuery/findAddress` | Already exists — extend to also return `addressCrossId` | `addressList[].addressId`, `addressList[].addressCrossId`, `latitude`, `longitude` |
   | `cross/findLocalityOnAddress` | Look up locality child of the address | `localityCrossId`, `localitySmwId` (nullable both) |
   | `cross/findBuilding` | Look up building child of the address | `buildingCrossId` |
   | `cross/createLocality` | POST CROSS `/v1/node/` with LOCALITY discriminator on the address | `localityCrossId`, `localitySmwId` |
   | `cross/findEdgeEquipmentCage` | Find free cage/pluggable on edge equipment; must also return the full EdgeEquipment descriptor | `cageCrossId` OR `edgeTransceiverCrossId`, `edgeEquipment` Map |
   | `cross/blockCage` | Mark cage or transceiver blocked and return parent locality cross id | `edgeLocalityCrossId` |
   | `cross/createOpticalPath` | POST CROSS `/v1/link/` (OpticalPath type) between building and edge locality | `opticalPathCrossId`, `opticalPathPrmId` |
   | `cross/createService` | POST CROSS `/v1/service/createFromServiceTemplate/` using `productId` as template | `serviceCrossId`; template expected to create the three components PRODUCT1, ACCESS, PHYSICAL_ACCESS |
   | `cross/addOpticalPathAndLocalityToService` | Attach OP + locality to the service-template components | — |
   | `cross/addCageToServiceComponent` | Attach cage/transceiver to PHYSICAL_ACCESS | — |

3. **Automaton `processes` array.** Add an entry so Automaton is
   allowed to start the process:

   ```json
   {
     "id": "GC-AUTOMATION.pniDesign",
     "container": "GC-AUTOMATION_1.0.0-SNAPSHOT",
     "authorities": ["AUTOMATON_WF_READ"]
   }
   ```

4. **Automaton `cross_wf_api.groovy` route.** Add a new route
   handler for `POST /gc-automation/v1/pniDesign` mirroring the
   existing `/resourceAvailabilityCheck` handler: validate against the
   yaml, `tmfService.createWithGeneratedId("GC-AUTOMATION", body)`,
   then `wfService.startWfSynchronousProcess("GC-AUTOMATION.pniDesign",
   [_order: result.id], null)`.

## Judgment calls / ambiguities

Points where the spreadsheet was ambiguous or silent and I chose
something — flag any that don't match your intent.

- **Row 6 "add speed into request"** — interpreted as a data element
  of the CROSS project payload. Not modelled yet; `cross/createProject`
  currently sends `name` + `projectType="GC_Fulfillment"` only.
  Decide whether `speed` belongs in the request body or in service
  attributes and I'll wire it in.

- **Row 9 "Find building"** originally had no substep letter. The
  corrected xlsx puts it at `3C.2`, which is how the BPMN orders it
  (after `findLocalityOnAddress`, before the locality gateway).

- **Rows 15-17 (PRODUCT1 / ACCESS / PHYSICAL_ACCESS)** — treated as
  descriptors of the **service-template components** created by
  `cross/createService` from `productId`, not as extra flow steps.
  The current BPMN does not itself create them one-by-one; it assumes
  the service template drives that.

- **Row 12 output `_EdgeLocalityCrossId`** — assumed `cross/blockCage`
  returns the parent locality of the edge equipment in the same
  response, so no extra lookup is needed.

- **`cageCrossId` OR `edgeTransceiverCrossId`** (rows 11, 12) — modelled
  as two separate process variables; `cross/findEdgeEquipmentCage`
  populates whichever one applies, and `cross/blockCage` /
  `cross/addCageToServiceComponent` get both passed through (the
  handler can ignore the null one).

- **Error handling beyond the address gateway** — only the "Not found"
  and "Not unique" address conditions are routed to the FAILED script.
  Failures in any other task currently propagate as runtime exceptions
  out of the SdcApiCall handler and surface as jBPM process errors.
  If you want more of the `PniDesignResponse` 400/404/409/422 codes
  (e.g. `MATERIAL_TEMPLATE_NOT_FOUND`, `PROJECT_NOT_CREATED`,
  `DESIGN_ALREADY_EXISTS`) to be translated into graceful FAILED
  responses, we'd add more gateways + condition expressions after the
  respective tasks.

- **Response variable naming** — the OpenAPI schema uses mixed case
  (`OpticalPathCrossId`, `CageCrossId`); the BPMN output keys preserve
  that exactly so the response validates. Internal process variables
  use lowercase (`opticalPathCrossId`). The mapping is done in the
  COMPLETED script task.

## Quick test after the Automaton side is wired up

1. `mvn clean install` — produces `target/GC-AUTOMATION-1.0.0-SNAPSHOT.jar`.
2. Deploy the kjar to the Kie Server container configured in
   Automaton's `processes` list.
3. Seed the request:

   ```bash
   curl -X POST http://<automaton>/api/dynamic/gc-automation/v1/pniDesign \
        -H 'Content-Type: application/json' \
        -u admin:admin \
        -d '{
              "version": "1.0",
              "addressMasterId": "1626222",
              "edgeEquipmentCrossId": "cAAA-AAss-AAEG",
              "serviceId": "SVC-0099",
              "crossProjectName": "ORD-1234",
              "productId": "PROD-FIBER-1G"
            }'
   ```
4. Observe `System.out.println` traces from the `on-entry` / `on-exit`
   scripts on each task (they report the key variables set, useful
   when bringing the Automaton sources up one at a time).
